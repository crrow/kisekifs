## Dependency notes

- 001 comes first because the current `main` CI run is red; later work needs a
  trustworthy remote gate rather than local-only evidence.
- 002 precedes 004 because `object_store` itself has an actionable advisory and
  its upgraded API should be the basis of the storage configuration work.
- 003 and 005 can run independently after 001.
- 006 starts only after 003–005 because it needs stable configuration, error
  contracts, and a green test harness before changing the durable protocol.
- 007 follows 006 because both touch `SliceBuffer`; it applies safe ownership
  to the new crash-consistent state machine instead of causing parallel drift.
- 008 requires the durability and safe-buffer contracts so mounted restart and
  concurrency assertions are meaningful.
- 009 uses the mounted suite as its signal/drain acceptance gate and takes
  ownership of the task/cache interfaces introduced by 006/007.
- 010 is offline by design and depends on 006 proving remote durability plus
  009 providing explicit per-volume cache roots and clean mount exclusion.

## Second production tranche

Plans 006–010 are the selected default top-five from the second audit. They
turn the previously vetted crash-consistency, unsafe, mounted-test, and
operational findings into self-contained handoffs. Operational lifecycle was
split: plan 009 owns running-mount resources and shutdown; plan 010 owns offline
backup/restore, because combining those failure domains would make review and
rollback unsafe.

Completing this tranche still does **not** by itself authorize a production-
ready claim. A later reconciliation must verify remote CI evidence, mounted
stress duration, restored-backup drills, supported workload requirements, and
the remaining items below.

## Remaining production work after plan 010

1. **Reference-aware object garbage collection and slice compaction.** Metadata
   deletion currently removes chunk-slice records without deleting immutable
   objects, and `write_slice` only notes compaction as a TODO. Design retention
   around open files, borrowed slices, and every retained metadata backup before
   deleting any object.
2. **Durable-format/schema evolution.** Bincode structures and RocksDB keys need
   explicit schema versions and forward/backward migration fixtures before an
   upgrade can be called safe.
3. **Long-running fault/stress and performance qualification.** Add scheduled
   multi-hour concurrent workloads, storage throttling/outage, cache pressure,
   repeated kill/recovery, leak checks, and workload-specific latency/SLO
   baselines after correctness gates are stable.
4. **Release and supply-chain operations.** Produce signed/reproducible Linux
   artifacts, SBOM/provenance, upgrade/rollback documentation, and an operator-
   tested release procedure.

## Direction grounded in the repository

- Treat Linux as the first production target. The justfile builds Linux images,
  the quick start uses Ubuntu/Lima, and FUSE CI currently lacks macFUSE. macOS
  can remain a later compatibility target instead of a permanently red required
  job.
- Define and version a **supported POSIX subset** before claiming compliance.
  The current README says the project is not compliant, while backend comments
  claim POSIX behavior; a machine-readable matrix prevents docs and code from
  drifting in opposite directions.
- Keep the existing `object_store`/OpenDAL split frozen while stabilizing the
  data protocol. Provider migration and crash-consistency changes in the same
  release would make failures impossible to attribute.

## Audit coverage and limits

Audited: workspace/manifests, CLI configuration and lifecycle, CI workflows and
the latest completed main baseline, metadata write/format/backup surfaces,
VFS/FUSE request and shutdown boundaries, write-back ownership/recovery,
dependency advisories, test topology, and all current `unsafe` blocks in
first-party storage/VFS Rust.

Not executed in the second audit: mounted filesystem tests, real S3/MinIO
failure injection, crash/restart fault matrix, backup/restore, performance
benchmarks, fuzzing, Miri, sanitizers, long-running stress, or Windows. The
first tranche's implementation evidence remains 112/112 local tests and green
CI through commit `2fb2c95`; the new areas remain unproven rather than
implicitly accepted.

## Findings considered and rejected

- Migrating all storage code from `object_store` to OpenDAL: rejected for this
  tranche because `AGENTS.md` freezes that abstraction and the migration would
  obscure reliability fixes.
- Treating every `unwrap()` in test code as a production bug: rejected; test
  assertions may unwrap. Production-call-path panics remain findings.
- The `users` advisory's supplementary-group bug as directly reachable through
  current code: rejected. KisekiFS only calls current uid/gid helpers, but the
  unmaintained direct dependency should still be removed in plan 002.
- Treating full POSIX compliance as the immediate target: rejected. A versioned
  supported subset with exact mounted tests is a safer release contract; add
  operations only when a target workload requires them.
- Uploading/copying every object into the metadata backup artifact: rejected.
  Plan 010 first proves metadata recovery against the same durable object store;
  object-store replication/versioning is an operator/storage concern and must
  be coordinated with future garbage collection.
