## Next production tranche

These were vetted but intentionally not mixed into the first five plans:

1. **Crash-consistent write protocol (P1, L, HIGH risk).** Make object upload,
   slice metadata publication, inode length, and staged-file deletion an
   explicit recoverable state machine. Add restart tests that terminate between
   every durable transition.
2. **Unsafe data-path containment (P1, M, HIGH risk).** Replace mutation through
   shared references, remove `Box::leak`, use disjoint safe slices for parallel
   reads, deny undocumented unsafe code, and add Miri/sanitizer coverage for the
   pool and slice buffer.
3. **Mounted acceptance suite (P1, L).** Define the supported POSIX subset and
   run fstests-style smoke, concurrent I/O, unmount/remount, ENOSPC, read-only,
   and injected object-store failure cases on Linux.
4. **Operational lifecycle (P2, L).** Add graceful shutdown/drain, health and
   readiness signals, bounded metrics cardinality, backup/restore verification,
   release artifacts, and an operator runbook.

## Direction grounded in the repository

- Treat Linux as the first production target. The justfile builds Linux images,
  the quick start uses Ubuntu/Lima, and FUSE CI currently lacks macFUSE. macOS
  can remain a later compatibility target instead of a permanently red required
  job.
- Define and version a **supported POSIX subset** before claiming compliance.
  The current README says the project is not compliant, while backend comments
  claim POSIX behavior; a machine-readable matrix prevents docs and code from
  drifting in opposite directions.
- Keep the existing `object_store`/OpenDAL split frozen while stabilizing the
  data protocol. Provider migration and crash-consistency changes in the same
  release would make failures impossible to attribute.

## Audit coverage and limits

Audited: workspace/manifests, CLI configuration and lifecycle, CI workflows and
the latest main run, metadata write/format paths, VFS/FUSE request boundaries,
write-back cache recovery behavior, dependency advisories, and all current
`unsafe` blocks in first-party Rust.

Not executed: full expensive test suite locally, mounted filesystem tests,
real S3/MinIO failure injection, crash/restart tests, performance benchmarks,
fuzzing, Miri, sanitizers, or Windows. Those areas remain unproven rather than
implicitly accepted.

## Findings considered and rejected

- Migrating all storage code from `object_store` to OpenDAL: rejected for this
  tranche because `AGENTS.md` freezes that abstraction and the migration would
  obscure reliability fixes.
- Treating every `unwrap()` in test code as a production bug: rejected; test
  assertions may unwrap. Production-call-path panics remain findings.
- The `users` advisory's supplementary-group bug as directly reachable through
  current code: rejected. KisekiFS only calls current uid/gid helpers, but the
  unmaintained direct dependency should still be removed in plan 002.
